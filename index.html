Validation-guided refinement

失败日志驱动的反馈闭环

tb-refine 不重新发散出题，而是在保留 source、archetype、assignment、canary 和 hidden oracle 边界的前提下，对现有 package 做定点修复。

1 · 真实验证

先跑静态 contract，再构建 environment/，分别执行 untouched baseline 和 reference solution。

2 · 选择证据

按失败阶段读取 build.log、baseline.log、solution.log 或 verifier.log。

3 · 约束修复

Agent 原地修复，但不能换题型、换 source，不能删除或弱化有意义的 oracle assertion。

4 · 重新过门

重新导出 public view 并再跑 Docker hard gates；默认最多两轮，逐轮保存 prompt、trace、日志与报告。

现在可以一条命令持续补产

researchfactory tb-pipeline ... --target-valid 100 --wave-size 20 --limit 160
该入口按 wave 执行 taskgen、反馈验证、来源记录和去重，达到目标后生成交付目录；原有独立命令仍可单独使用。

Diversity control

不再只是 repair / fix

assignment 是只读的隐藏契约；blueprint 与 manifest 必须保持一致。非 repair 类型的 baseline failure 来自目标产物尚未创建，而不是人为注入故障。

01 · troubleshoot_repair

故障排查与修复

诊断已有代码、配置或数据中的真实缺陷。

02 · implement_feature

功能实现

为可用项目增加缺失的 CLI、服务或库能力。

03 · configure_service

服务配置

创建满足明确运行要求的工具或服务配置。

04 · build_package

构建与打包

产出可安装软件包、二进制或可复现工件。

05 · transform_analyze_data

数据处理与分析

把真实输入转成数据集、索引或结构化报告。

06 · security_audit_hardening

安全审计与加固

评估安全状态并产出可验证的策略与加固结果。

07 · deploy_integrate

部署与集成

连接组件，形成可检查的本地服务或 CLI 行为。

08 · migrate_upgrade

迁移与升级

迁移代码、依赖、配置或数据格式。

09 · optimize_performance

性能优化

在保持正确性的同时改善可测量性能指标。

10 · investigate_report

调查与报告

从日志或系统状态产出基于证据的诊断报告。

11 · author_tests_validation

测试与验证

创建可复用 checker、linter 或测试套件。

12 · automate_workflow

工作流自动化

实现幂等、可重复运行的多步骤脚本或流水线。

Quality boundary
